# Module 16: Capstone Project 🚀

Your final project! Build, deploy, and present your own app.

## What You'll Build

A working web app that **you** designed. It must:
- Solve a real problem or be fun to use
- Have at least 2 tables in the database (data that connects)
- Let users sign up and log in
- Be deployed live on the internet
- Have code saved on GitHub

**It's YOUR project** — not a copy of what you learned in the modules. Build something you actually want to use!

## Requirements (what we're checking)

- ✅ **It works** — all the main features work end-to-end
- ✅ **Data storage** — a database with at least 2 connected tables
- ✅ **Users & login** — people can sign up, log in, and see their own data (not other people's)
- ✅ **Deployed** — a real URL on the internet where anyone can use it
- ✅ **GitHub repo** — your code is saved with a good history of changes
- ✅ **Tests** — you tested the important parts to make sure they work
- ✅ **Handles problems** — if something breaks, the app doesn't crash (good error messages)
- ✅ **Looks good** — clean design that works on phones
- ✅ **Everyone can use it** — good contrast (readable text), keyboard navigation works
- ✅ **You understand it** — you can explain every decision and every piece of code
- ⭐ **Bonus: Automation** — if you did Module 13, show off some automated deployment or testing

## Project Ideas

Pick one of these or your own (just ask an instructor):

**Easy level:**
- Pet tracker 🐕 (add pets, set care reminders, track their birthday)
- Movie/game list 🎬 (save movies/games you want to watch/play, rate them, review them)
- Habit tracker 📊 (check off daily habits, see your streak)
- Recipe saver 🍕 (save favorite recipes, organize by meal type)

**Medium level:**
- Allowance tracker 💰 (track spending, set goals, see categories)
- Study buddy 📚 (list subjects/topics, quiz yourself, track progress)
- Game tournament tracker 🏆 (organize brackets, track scores, see leaderboards)
- Book review site 📖 (post reviews, rate books, recommend to friends)

**Advanced level:**
- Social media clone (follow friends, post updates, comment, like)
- Multiplayer game tracker (track games with friends, see win/loss records)
- Team project planner (assign tasks, set deadlines, track progress)

## Governance & Understanding (You Direct Claude Code)

Before you submit, you need to show that you understand and own your project. Do this:

### Step 1: Tell Claude Code to generate project governance

Open Claude Code and say something like:

```
Generate a CLAUDE.md for my [project name] app. 

Include:
- How I used Cursor and Claude Code (what I directed them to do, and how I verified their work)
- My rules for testing (what must be tested before I call it done)
- My rules for deploying (what checks must pass before shipping to Vercel)
- How I kept secrets safe (where API keys live, never in my code)

Make it sound like me, like I'm explaining my project to a friend.
```

Claude Code will generate a draft CLAUDE.md. Read it. Change anything that's wrong. Add anything you missed. **This is YOUR project's rules — make it true.**

### Step 2: Tell Claude Code to log your decisions

Say something like:

```
Generate a decisions.md for my [project name] app.

List the 3-4 biggest choices I made:
- Why I picked [framework/tool/approach]
- What other things I considered
- Why I didn't pick those things

Make it sound like I actually thought about these decisions.
```

Claude Code will generate decisions.md. Read it. Make sure it's true. Fix anything that's wrong. **This is YOUR decision log.**

### Step 3: Submit both files

When you submit your capstone, include CLAUDE.md and decisions.md alongside your code and video.

**Why?** These files prove you understood what you built, not that you just copied something. They show that you made deliberate choices and can explain them. That's what matters.

## What To Submit

1. **Live URL** — a working link to your app (so we can use it)

2. **GitHub repo** — your code with commits and a README
   - README should explain: what your app does, how to run it locally, the main tools you used

3. **CLAUDE.md** — your project's rules (generated by Claude Code, reviewed and owned by you)
   - How you used Cursor/Claude Code
   - Rules for testing, deployment, secrets
   - Any automations you set up

4. **decisions.md** — the big choices you made (generated by Claude Code, reviewed and owned by you)
   - 3-4 major decisions
   - Why you picked each one
   - What alternatives you considered

5. **One-page write-up** — the story of your project
   - What it does
   - The stack you picked and why (mention 1-2 alternatives you considered)
   - The decisions you're most proud of and why they matter

6. **Video walkthrough** — 3-5 minutes where you demo your app and explain one key decision
   - Show us using your app
   - Pick one decision from decisions.md and explain why you made it
   - Show how you used Cursor or Claude Code (the actual tool — not just that you used it, but HOW you directed it)

## How It's Graded

Scored on the rubric below — 10 core things we check. **Pass = "Good Enough ✅" or "Awesome! 🌟" on everything + 80% total score.**

| What We're Checking | Not Good (0) | Getting There (1) | Good Enough ✅ (2) | Awesome! 🌟 (3) |
| --- | --- | --- | --- | --- |
| **Does It Work?** | App crashes or won't run | Runs, but main features broken | All features work as you said | Everything works + extra features |
| **Database & Security** | No database or auth doesn't work | Data works but anyone can see anyone's data | Database works, users can only see their own stuff | Smart database design, secure, handles weird cases |
| **Code Quality** | Code is messy/unorganized | Works but hard to read | Clean, organized, easy to understand | Really well-structured and documented |
| **GitHub Repo** | No repo or just one commit | Repo exists but bad history | Good commit messages, shows your work | Excellent history, PRs, collaboration-ready |
| **Deployed & Live** | Not on the internet | Deployed but broken | Works on Vercel with auto-deploy | Live + preview deploys + custom domain |
| **Your AI Workflow** | Doesn't show how AI was used, or CLAUDE.md/decisions.md missing | Used AI randomly; CLAUDE.md/decisions.md vague or generic | Deliberate use of Cursor/Claude Code (CLAUDE.md shows rules you set); decisions.md shows actual choices; video shows you directing the tools; can explain why you used AI for certain tasks | Sophisticated, well-documented workflow; CLAUDE.md + decisions.md prove you thought deeply about your choices; video shows mastery of directing the tools; examples of verification/review |
| **Testing & Errors** | No testing, crashes on errors | Some testing, errors not handled | Key features tested, errors show nice messages | Strong testing, handles edge cases, really robust |
| **Design & UX** | Looks broken/unfinished | Functional but looks like AI made it | Clean design, looks professional, works on phones | Polished, beautiful, great on all screen sizes |
| **Accessibility** | Can't use keyboard, text hard to read | Missing some labels/contrast | Keyboard works, good contrast, readable | Really accessible, passes audits |
| **Can You Explain It?** | Can't explain your own code | Explains some things, weak on "why" choices | Clearly explains major decisions (decisions.md backs this up); shows you considered alternatives; video + written docs prove you understand tradeoffs; you own every piece | Could teach someone else how/why you built it; decisions.md shows deep thinking; CLAUDE.md proves you set guardrails; can explain even the tricky parts |
